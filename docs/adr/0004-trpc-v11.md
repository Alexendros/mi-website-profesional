# ADR-0004 · tRPC v11 como API layer type-safe

- Fecha: 2026-05-10
- Estado: Aceptado
- Decisor: Alexendros

## Contexto

El hub `alexendros.pro` necesita una API layer entre Server Components, Client Components y el backend (Prisma + Supabase). El contrato de tipos debe ser end-to-end sin codegen manual.

## Decisión

**tRPC v11** con `fetchRequestHandler` para App Router. **`@tanstack/react-query v5`** como peer dependency para client-side caching.

## Justificación

| Criterio | tRPC v11 | REST (Next.js API routes) | GraphQL/Apollo |
|---------|:---:|:---:|:---:|
| Type-safety E2E | ✅ sin codegen | ✗ (manual) | ✅ con codegen |
| Coste de setup 1 dev | bajo | muy bajo | alto |
| Server Components caller | ✅ `createCallerFactory` | ✅ | parcial |
| Streaming SSE | ✅ | manual | subscriptions |
| Overhead bundle | mínimo | mínimo | alto |
| Documentación ES | buena | amplia | amplia |

## Consecuencias

- `fetchRequestHandler` en `app/api/trpc/[trpc]/route.ts` — no `createNextApiHandler` (Pages Router, descatalogado).
- `createCallerFactory` para llamar procedures desde Server Components sin HTTP round-trip.
- Rate limiting en `/api/trpc/*` con Upstash Redis — objetivo <200ms p95 sin DB fría.
- React Query v5 como peer dependency: `@tanstack/react-query@5`.

## Alternativas descartadas

- **REST puro**: type-safety manual es carga cognitiva alta; errores en runtime difíciles de detectar en monorepo con múltiples apps.
- **GraphQL**: overhead de codegen y esquema no justificado para 1 dev. Reconsiderar si el equipo crece >3 devs.
